6.1 Front End vs Back End: The Full Picture

Everything so far ran in the browser — the front end, the part users see. The back end is the program running on a server somewhere that the front end talks to: it owns the database, enforces the rules, protects the secrets, and answers requests. In Part 5 you consumed other people’s APIs; in this part you build your own. Node.js makes this possible in JavaScript — the same language, new superpowers: reading files, opening network ports, talking to databases.

6.4 REST APIs: The Grammar of the Web

A REST API maps the four things you do with data — Create, Read, Update, Delete (CRUD) — onto four HTTP methods. Every modern app speaks this grammar:

HTTP MethodCRUD ActionExample RouteMeaning
GETReadGET /api/jobsGive me all jobs
GETRead oneGET /api/jobs/3Give me job number 3
POSTCreatePOST /api/jobsHere is a new job — store it
PUTUpdatePUT /api/jobs/3Replace job 3 with this version
DELETEDeleteDELETE /api/jobs/3Remove job 3

STATUS CODES YOU MUST KNOW

200 OK · 201 Created · 204 No Content · 400 Bad Request (their fault) · 401 Unauthorized · 404 Not Found · 500 Server Error (your fault). Returning the right code is a mark of professionalism. For more comfortable testing, install the free Postman app or the Thunder Client extension in VS Code.

6.6 Serving Your Front End From Your Back End

One line lets Express serve your HTML/CSS/JS files: create a folder named public, put a front end inside, and add app.use(express.static("public")); near the top of your server. Now http://localhost:3000 serves your page AND your API — one app, front to back. This is the architecture of the Part 7 capstone.

Try This: Add a route GET /api/jobs/stats returning { total, averageSalary, byType } computed with filter and reduce. Careful: define it BEFORE the /api/jobs/:id route, or Express will think "stats" is an id — your first real routing lesson.
