9.1 Git & GitHub: The Developer’s Time Machine

Git records snapshots ("commits") of your project so you can travel back in time, work fearlessly, and collaborate. GitHub hosts your Git projects online — and doubles as your public proof of skill. Recruiters genuinely look at it. No professional developer works without Git; it appears on essentially every job description.

COMMIT MESSAGE DISCIPLINE

Write messages that complete the sentence "This commit will…": "Add salary validation", "Fix delete button bug". Your commit history becomes a professional diary that interviewers actually read.

CommandWhat It Does

git checkout -b feature-xCreate a parallel timeline (branch) to experiment safely
git merge feature-xBring a branch’s work into your main timeline
git clone URLDownload any public project to learn from it

9.2 Debugging Like a Professional

Professionals are not people who write bug-free code; they are people who find bugs fast. Follow the DIRE method: Describe the error (read the message, top line first — it names the file and line number), Isolate the smallest failing piece, Replicate it reliably, then Experiment with one change at a time.

  • Console tactics: console.log({ variableName }) (braces print the name AND value), console.table(arrayOfObjects) for beautiful tabular output, console.error() for failures.
  • Breakpoints: in Chrome DevTools → Sources tab, click a line number; execution pauses there and you can inspect every variable, then step line by line. Strictly better than log-spamming for tricky bugs.
  • The rubber-duck method: explain your code, line by line, out loud, to an object on your desk. The bug routinely reveals itself mid-sentence. Every professional does this; few admit it.

The Five Errors Every Beginner Meets

Error MessageAlmost Always Means
Uncaught TypeError: Cannot read properties of nullYour selector found nothing — typo in the id/class, or the script ran before the HTML loaded
ReferenceError: x is not definedMisspelled variable name, or using it before declaring it
SyntaxError: Unexpected tokenA missing/extra bracket, parenthesis, or quote nearby
Failed to fetch / CORS errorServer is not running, wrong URL/port, or the server has not allowed your origin (see Part 8.2)
undefined is not a functionCalling something that is not a function — often a typo like .lenght() or calling a property

9.3 Deploying: Putting Your Work on the Real Internet

Static Sites (Parts 1–5 projects) — Netlify, free

  1. Push the project folder to GitHub (Section 9.1).
  2. Create a free account at netlify.com and choose "Import from GitHub".
  3. Select the repository, click Deploy. In about a minute you get a live URL like your-site.netlify.app — and every future git push redeploys automatically.

Full-Stack Apps (Parts 6–7) — Render, free tier

  1. In package.json, add a start script: "scripts": { "start": "node server.js" }, and make the port flexible: const PORT = process.env.PORT || 3000;
  2. Push to GitHub. Create a free account at render.com → New → Web Service → connect the repo.
  3. Build command: npm install. Start command: npm start. Deploy — your API and front end go live at a public URL.
NOTE ON FREE TIERS

Free services sleep after inactivity (first visit may take ~30 seconds) and a SQLite file may reset on redeploy — perfectly fine for a portfolio. Production apps graduate to a hosted PostgreSQL database, and your Part 7 SQL transfers almost unchanged.

9.4 Your Portfolio and the Job Hunt

The Portfolio That Gets Interviews

  • Three to five deployed projects with live links AND GitHub links: your styled profile site, weather app or GitHub finder, the to-do or quiz app, and — front and centre — the full-stack Job Board capstone.
  • A README for every repository: what it does, a screenshot, the technologies used, how to run it locally, and one thing you found hard and solved. That last item is what interviewers ask about.
  • Green squares: commit code most days. A consistent GitHub activity graph quietly screams discipline.

Positioning Yourself (especially for remote roles)

  1. CV headline: "Junior Full-Stack JavaScript Developer — HTML · CSS · JavaScript · Node.js · Express · SQL · Git". List projects above education; describe each with verbs and outcomes ("Built a full-stack job board with search, filtering and a SQLite database; deployed on Render").
  2. LinkedIn: same headline, featured section linking your live capstone, and one short post per week about what you built or learned. Documenting your journey in public attracts recruiters.
  3. Apply in volume with focus: junior/graduate front-end or full-stack JS roles, internships, and paid freelance gigs (small business sites are an underrated first income). Ten tailored applications beat a hundred generic ones.
  4. Interview preparation: re-do FizzBuzz, the array Big Three (map/filter/reduce), explain the event loop with the restaurant analogy, and be ready to walk through your capstone code line by line — because you wrote every line, you can.
  5. Keep growing: after this guide, the natural next steps are TypeScript (JavaScript with safety rails) and React (the dominant front-end framework). Your State → Render → Actions pattern from Part 4 means React will feel familiar, not foreign.

9.5 Your Daily Professional Habits, From Today

  • Code 90+ minutes daily; commit and push what you build.
  • Read error messages fully before searching for solutions.
  • When stuck for more than 30 minutes, rubber-duck it, then search the exact error message, then ask for help with a minimal example.
  • Rebuild one old project from memory each month — fluency comes from repetition, not exposure.
  • Use official documentation (MDN Web Docs at developer.mozilla.org is your bible) over random blog posts.

A.4 Recommended Free Resources

  • MDN Web Docs (developer.mozilla.org) — the definitive HTML/CSS/JS reference.
  • The Odin Project (theodinproject.com) — a free full-stack curriculum to reinforce this guide.
  • roadmap.sh — visual maps of what to learn next for any developer role.
  • JavaScript.info — the best free deep-dive book on modern JavaScript.
  • freeCodeCamp — thousands of practice exercises and free certifications.

A suggested rhythm for each study day (2–4 hours). Adapt freely, but protect the building time — it is where the learning actually happens.

SegmentDurationActivity
Warm-up15 minRetype yesterday’s key practical from memory
New material45–60 minRead the next section; run every copy-and-paste practical
Build60–90 minDo the Try This challenges and project work — no copying, struggle is the point
Consolidate15 minCommit to GitHub; write 3 lines in a learning journal: what I built, what broke, what I learned

Weekly rhythm: five study days, one project-only day (no new material — just build), one full rest day. Rest is not optional; your brain consolidates code while you sleep.
